# 먹자 골목 관리

먹자 골목 관리는 식당 요청 작성, 부서별 요청 확인, 입고 확인, 요청 내역 조회, 메뉴/레시피 관리, CSV 가져오기/내보내기를 처리하는 웹앱입니다.

현재 버전은 **간편모드 중심**입니다. 별도로 일반모드 개발을 시작하기 전까지 품목 행과 요청 화면은 간편모드 기준으로 관리합니다.

## 접속 주소

Netlify 배포 주소:

```text
https://mukjamtl.netlify.app
```

개발 중 화면 확인도 기본적으로 Netlify 배포 주소에서 진행합니다.

## 입장 비밀번호

회원가입 없이 입장 비밀번호로 권한을 구분합니다.

| 역할 | 비밀번호 | 주요 권한 |
| --- | --- | --- |
| 카페테리아 | `c1234` | 카페테리아 요청 확인, 메모 작성 |
| 야채 | `v1234` | 야채 요청 확인, 메모 작성 |
| 그로서리 | `g1234` | 그로서리 요청 확인, 메모 작성 |
| 레스토랑 | `m1234` | 전체 요청 작성, 입고 확인, 메뉴/레시피 조회 |
| 관리자 | `madmin` | 전체 기능, 비밀번호 관리, CSV 관리 |

모바일 브라우저의 비밀번호 저장 기능을 사용하면 반복 로그인 부담을 줄일 수 있습니다.

## 화면 구성

### 홈

홈에서는 로그인한 역할에 따라 다른 내용을 보여줍니다.

카페테리아, 야채, 그로서리 담당자는 자기 부서로 들어온 최근 요청을 확인하고 메모를 작성할 수 있습니다.

레스토랑과 관리자는 전체 부서의 최근 요청과 입고 현황을 확인하고, 품목별 입고 체크를 저장할 수 있습니다.

메모는 요청별로 저장됩니다. 로그인한 부서는 자기 메모를 수정할 수 있고, 다른 부서 메모는 읽기 전용으로 표시됩니다. 메모가 없으면 `메모 없음`으로 표시됩니다.

### 요청 내역

요청 내역은 화요일부터 일요일까지 주 단위로 조회합니다.

페이지 이동은 `<<`, `<`, 번호 5개, `>`, `>>` 형식입니다. 데이터가 없는 번호 칸은 링크 없이 표시됩니다.

각 행을 선택하면 상세 내역 화면으로 이동하며, 상세 내역은 부서별 카드와 품목별 출고/입고 체크 열로 표시됩니다. 체크 수정은 관리자 권한에서만 가능합니다.

### 요청하기

요청하기 화면은 레스토랑과 관리자만 사용할 수 있습니다.

주문 모드에서는 DB에 등록된 모든 재료를 부서별, 카테고리별로 확인하고 필요한 품목을 체크한 뒤 메모와 함께 요청을 저장합니다.

수정 모드는 관리자만 사용할 수 있습니다. 수정 모드에서는 재료 품목과 카테고리를 관리하고, 선택한 품목의 부서/카테고리 변경과 순서 이동을 할 수 있습니다.

현재 주문 부서는 다음 세 가지입니다.

- 카페테리아
- 야채
- 그로서리

카페테리아 기본 카테고리는 반조리, 반찬, 소스, 냉장, 냉동입니다. 야채는 기본적으로 야채 카테고리를 사용하고, 그로서리는 상온, 냉장, 냉동, 기타 카테고리를 사용합니다.

간편모드에서는 품목 행에 품목명만 표시하고, 주문 단위는 DB에는 저장하지만 화면에는 표시하지 않습니다.

### 메뉴

메뉴 화면은 레스토랑과 관리자만 사용할 수 있습니다.

메뉴는 카테고리별로 표시되며, 검색과 카테고리 선택으로 필터링할 수 있습니다. 판매 중단 메뉴는 메뉴명에 취소선을 표시합니다.

관리자는 메뉴 카테고리별로 메뉴를 추가하고, 각 메뉴 행에서 수정하거나 삭제할 수 있습니다. 메뉴를 추가하면 연결된 빈 레시피도 함께 생성됩니다.

메뉴 행의 레시피 아이콘을 누르면 연결된 레시피가 팝업으로 표시됩니다.

### 레시피

레시피 화면은 모든 로그인 사용자가 조회할 수 있습니다.

관리자는 레시피의 설명, 재료, 조리 순서, 메모를 수정할 수 있습니다. 재료와 조리 순서는 항목별로 추가, 수정, 삭제할 수 있고 순서 이동 핸들러를 제공합니다.

레시피 삭제는 확인 절차를 거쳐 진행됩니다.

### 관리자

관리자 페이지는 관리자 권한으로만 사용할 수 있습니다.

관리자 페이지의 현재 기능은 다음 두 가지입니다.

- 비밀번호 관리
- CSV 관리

비밀번호 관리는 리스트 형식이며, 사용자 추가는 제목 오른쪽 `+` 원 아이콘으로 진행합니다. 기존 항목 수정은 행의 수정 아이콘을 눌러 폴딩 폼에서 처리합니다. 삭제 버튼은 항상 수정 폼 안에만 표시합니다.

CSV 관리는 아래 네 가지 데이터를 가져오거나 내보낼 수 있습니다.

| 항목 | 내용 |
| --- | --- |
| 요청 내역 | 저장된 요청/입고 내역 |
| 재료 목록 | 요청하기 화면에서 CRUD 하는 재료 DB |
| 메뉴 | 메뉴 DB |
| 레시피 | 레시피 DB |

CSV 가져오기는 기존 데이터를 교체하므로 확인 창을 거친 뒤 실행됩니다.

## 데이터 저장 방식

현재 버전은 **DB 우선 저장 + localStorage 보조 저장** 구조입니다.

Netlify DB 환경변수가 연결된 배포 환경에서는 요청, 입고 확인, 메모, 재료, 메뉴, 레시피, 비밀번호 관리 데이터가 PostgreSQL DB에 저장됩니다.

DB 연결이 없거나 Netlify Functions가 동작하지 않는 환경에서는 브라우저 `localStorage`를 보조 저장소로 사용합니다. 이 경우 다른 기기나 다른 브라우저와 데이터가 공유되지 않습니다.

현재 DB는 PostgreSQL 기준이며 아래 테이블을 사용합니다.

| 테이블 | 목적 |
| --- | --- |
| `access_identities` | 브라우저별 익명 사용자, 역할, 부서 기록 |
| `access_logs` | API 접속 IP, 기기, 역할 로그 |
| `access_accounts` | 입장 비밀번호, 역할, 부서, 표시 이름, 확장용 사용자 이름 |
| `app_settings` | 부서별 요청 카테고리 등 설정 |
| `orders` | 요청 날짜, 시간, 메모, 요청 메시지 |
| `order_items` | 요청별 품목, 카테고리, 부서, 입고 상태 |
| `order_memos` | 역할/부서별 요청 메모 |
| `receipt_confirmations` | 입고 확인 기록 |
| `department_confirmations` | 추후 부서별 출고 확인 기록 |
| `recipes` | 레시피 본문, 재료, 조리 순서, 메모, 이미지 URL |
| `ingredients` | 요청하기 화면의 재료 품목 DB |
| `menus` | 메뉴 한글/영문 이름, 카테고리, 계절 메뉴, 판매 중단, 가격, 연결 레시피 |

접속 IP와 브라우저 정보는 화면에 표시하지 않고, 나중에 로그인/회원 기능과 감사 기록을 붙일 때 사용할 수 있도록 서버에 기록합니다.


## 테스트 방식

현재 개발 중 화면 테스트는 로컬 서버가 아니라 GitHub `main` 브랜치와 Netlify 배포 페이지를 기준으로 합니다.

개발자는 변경 사항을 커밋하고 GitHub에 push한 뒤, Netlify 자동 배포가 끝난 배포 페이지에서 화면을 확인합니다.

페이지 업데이트가 늦게 보이면 아래 캐시 초기화 페이지를 먼저 엽니다.

```text
https://mukjamtl.netlify.app/reset-cache.html
```

Netlify 배포 반영은 push 후 잠시 지연될 수 있습니다. 확인 기준은 `service-worker.js`의 캐시 버전과 HTML의 JS/CSS 버전 쿼리입니다.

자동 사용자 흐름 테스트:

```bash
npm run test:user-flows
```

## 배포 방식

이 프로젝트는 정적 웹앱과 Netlify Functions를 함께 사용합니다.

Netlify 설정은 `netlify.toml`에 들어 있으며, 현재 publish directory는 저장소 루트입니다.

```toml
[build]
  publish = "."
  command = ""

[functions]
  directory = "netlify/functions"
```

GitHub `main` 브랜치에 변경 사항을 push하면 Netlify가 자동으로 배포합니다.

## Netlify DB 연결 확인

DB 연결에 사용하는 값은 저장소에 직접 넣지 않습니다. Netlify UI에서 Functions scope 환경변수로 설정해야 합니다.

지원하는 환경변수 이름은 아래 순서로 확인합니다.

- `NETLIFY_DATABASE_URL`
- `NETLIFY_DATABASE_URL_UNPOOLED`
- `DATABASE_URL`
- `POSTGRES_URL`
- `NEON_DATABASE_URL`

DB 상태 확인 API:

```text
https://mukjamtl.netlify.app/api/health
```

정상 연결이면 아래와 같이 응답합니다.

```json
{"ok":true,"db":true}
```

## V1.0 전 점검

실제 데이터 입력 전에는 데모 데이터 확인을 마친 뒤 DB 초기화를 진행하고, 재료 목록, 메뉴, 레시피, 요청 내역을 운영 데이터 기준으로 다시 입력합니다.

초기화는 데모 이후 별도 작업으로 진행합니다. 이 문서 업데이트 과정에서는 DB 데이터를 변경하지 않았습니다.
